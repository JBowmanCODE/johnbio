## 1. SEO

### Issue: live `sitemap.xml` does not match source

The source sitemap uses clean URLs and proper differentiated priorities (homepage 1.0, top tools 0.9, course pages 0.7, legal 0.5). The live sitemap returns `.html` extensions and uniform `priority=0.8`. This is a **deploy drift**, not a code bug.

**Why it matters:** `.html`-suffixed sitemap URLs all 301-redirect to clean URLs (per `.htaccess` line 16). Google and AI retrieval crawlers waste crawl budget chasing the redirect chain. Search Console will show "Page with redirect" or "Alternate page with proper canonical" warnings.

**Recommended action:**
- Check the GitHub Actions FTP-deploy workflow at `.github/workflows/` — confirm `sitemap.xml` is in the deploy file list, not in the exclude list.
- After confirming, push a no-op change to `sitemap.xml` (touch the comment) and verify the live URL updates within a few minutes.
- If the deploy works but the live file still drifts, look for a hosting-side script regenerating sitemap.

### Issue: homepage `ItemList` schema is stale

`index.html` line 76 declares `"numberOfItems": 19` and the `itemListElement` array has 19 items. But the site has at least 22 production tools now (Football 2026, T&Cs Simplifier, Prompt Improver, FAQ Schema Generator, EU AI Act, plus the original 19). The array also references `Zig Zag Drop Slot` at position 15 but doesn't include the four newer tools.

**Recommended action:** Regenerate the `itemListElement` array from the same source as `index.js` `TOOLS` so they cannot drift. Bump `numberOfItems` to match.

### Issue: `news/index.html` ships a thin `Blog` schema with no posts

Line 42–55: only a top-level `Blog` object, no embedded `ItemList` of posts. Article pages have full schema; the index page doesn't enumerate them.

**Why it matters:** GEO/AEO retrieval prefers index pages that explicitly list their items as schema-typed entries. This is the difference between "JohnB writes about AI" and "JohnB published these five articles".

**Recommended action:** Generate an `ItemList` JSON-LD block from `news-data.js`'s `NEWS_POSTS` at build time, or hand-maintain it for now. Each post becomes an `Article` `ListItem`.

### Issue: `news.css` version string still missing on news index

Line 40: `<link rel="stylesheet" href="/news.css">` — no `?v=` suffix. Article pages load `/news.css?v=4`. Anyone hitting `/news` after a CSS change keeps the stale cached file until natural cache expiry.

**Recommended action:** Change to `<link rel="stylesheet" href="/news.css?v=4">`.

### Issue: `robots.txt` blocks pages that no longer need blocking

Source `robots.txt` still lists `Disallow: /header.html` and `Disallow: /footer.html`. With `.htaccess` 301-redirecting any `.html` request, these URLs are unreachable directly anyway. The `Disallow` is harmless but redundant.

`michael-scott.html`, `heart.html`, `template.html` are still in `Disallow` and the source does have these files. If they're meant to be private, that's fine. If they were temporary, consider deleting them.

**Recommended action:** Remove `/header.html` and `/footer.html` from `robots.txt` (the .htaccess redirect makes them unreachable). Confirm the others are intentionally kept.

### Quick wins

- Bump `index.html`'s `<title>` from `JohnB.io | AI Projects Gallery` to `John Bowman - AI Tools, Free LLM Apps and AI Course | JohnB.io` for keyword payload. The page is functionally a tools directory plus career front door.
- Standardise title separator across all pages (mix of `|` and `-` still present — `index.html` uses `|`, `eu-ai-act.html` uses `-`, etc.).
- `index.html` `<meta property="og:url">` is `https://johnb.io` (no trailing slash) but canonical is `https://johnb.io/` (with). Pick one. Recommended: keep canonical as `https://johnb.io/` and update OG URL to match.

## 4. Site Optimisation & Modern Web Practices

### Issue: header and footer still fetched at runtime

`scripts.js` `loadHeaderAndFooter()` continues to `fetch('/header.html')` and `fetch('/footer.html')` after DOMContentLoaded. The site uses `sessionStorage` caching to avoid the flash on subsequent navigations, which is clever — but the first page view still costs two round-trips.

For a static site with GitHub Actions FTP deploy, the simplest fix is a build step that inlines header.html and footer.html into each HTML file at deploy time. A 30-line Node script reading the partials and substituting `<div id="header-placeholder"></div>` would do it. Optional improvement, not urgent.

### Issue: pdf.js, jspdf, docx, mammoth versions

Reading `aitocv.html` confirmed:

```
mammoth@1.8.0          → current 1.10.x+
pdf.js v4.11.246       → current 5.6.x
jspdf 2.5.1            → current 4.0.x
docx@7.8.2             → current 9.x+
```

The jspdf version is affected by **CVE-2025-68428** (path traversal in font loading, fixed in 4.0.0). For a browser-side library the practical exploitability is lower than the Node case, but still worth bumping.

The unpkg `docx@7.8.2` script has no SRI and no `crossorigin` attribute. If unpkg gets compromised, arbitrary script runs on `johnb.io`.

**Recommended action:** Bump pdfjs-dist to 5.6.x, jspdf to 4.0.x, mammoth to 1.10.x, docx to a version pinned with SRI. Re-test the upload-CV flow on `/aitocv` after each upgrade.

### Issue: `get_captions.php` is the only PHP file on the site

Source has a `get_captions.php` (1,082 bytes) at root. The `error_log` shows it was used for YouTube transcript fetching at one point. Looking at it again given the YouTube API key leak, this PHP file is the source of the leak — it embedded the key in the URL it fetched, which then ended up in `error_log`.

**Recommended action:**
1. If `get_captions.php` is no longer in active use (the existing `/transcripts` tool seems to expect users to paste transcripts manually), delete it.
2. If it is still in use, refactor to read the API key from an environment variable (Namecheap supports SetEnv in `.htaccess`), not embed it in the URL. Move the key into `.env` (already in `.gitignore`).

- HTML correctly set to no-cache must-revalidate.

## 5. Security Issues

### CRITICAL: leaked YouTube Data API key in `error_log`

`error_log` at the repo root (8 entries, 8 July 2024 12:54 UTC) contains the same URL repeated:

```
https://www.googleapis.com/youtube/v3/captions/AUieDaYHTUZJoM2eGQhnVb8X1kBqzDtc9RiFiHUCm64jmgJ0eaE?key=REDACTED
```

`.htaccess` blocks public HTTP access to the file (lines 130–134), so a random scraper can't pull `https://johnb.io/error_log` over the web. **But** `error_log` is NOT in `.gitignore` (`.gitignore` covers `workers/`, `.env*`, `.DS_Store`, `Thumbs.db`). If the deployment Git repo at `github.com/JBowmanCODE` is public, the API key is in the commit history.

**Why it matters:** Google API keys are persistent until rotated. A leaked key with YouTube Data API enabled lets an attacker burn through your daily quota (10,000 units/day default), trigger overage billing on Cloud projects with billing enabled, or pivot if the key has broader scope.

**Recommended action (in this order):**
1. **Today.** Rotate the key in Google Cloud Console → APIs & Services → Credentials. Create a new key, restrict it to YouTube Data API and the `johnb.io` HTTP referrer, then delete the old key.
2. **Today.** Add to `.gitignore`:
   ```
   error_log
   *.log
   ```
3. **Today.** Run `git rm --cached error_log && git commit -m "Stop tracking error_log"`. The file stays on the deployed server (Apache writes to it), but git stops tracking it.
4. **Today.** If the GitHub repo is public, also use `git filter-repo` or the BFG Repo-Cleaner to strip historical occurrences of the key from git history. Force-push.
5. **Later this week.** Investigate `get_captions.php` — see §4. The PHP file embedding the API key into a URL is the root cause of the leak.

### Issue: `firebase-init.js` exposes Firebase web config

`firebase-init.js` lines 10–18 expose:

```
apiKey: "AIzaSyBBZeNnPzS3cNGzuoIZNvB5oiVUn0nd5a4"
authDomain: "johnblogin-fdcc6.firebaseapp.com"
projectId: "johnblogin-fdcc6"
storageBucket: "..."
messagingSenderId: "..."
appId: "..."
measurementId: "G-XDR9MJ7S6E"
```

**This is normal and expected for Firebase Web SDK** — the API key is a public identifier, not a secret. Security depends entirely on Firestore Security Rules and (optionally) Firebase App Check.

**Recommended action (verify in Firebase Console, not the code):**
1. **Firestore rules** — confirm rules require authenticated users for all writes and limit reads appropriately. The default rules are wide-open for 30 days then closed. If your project is older than 30 days and you haven't customised, you're locked out — fine. If you have customised, double-check.
2. **App Check** — enable Firebase App Check with reCAPTCHA Enterprise or Cloudflare Turnstile. This blocks API calls from anywhere except your own domain.
3. **Authorized Domains** — under Authentication → Settings, restrict authorized domains to `johnb.io`, `localhost`, and any preview domains you actually use.
4. **API key restrictions** — in Google Cloud Console → Credentials, restrict the Firebase Web key to the Firebase service APIs only and to HTTP referrers ending `johnb.io`.

The Firebase API key is not the same as the YouTube API key in §5.1 — different rotation, different consequences. Both worth checking.

## 7. General Improvements

### Idle hero animation only fires on mouse-move

`index.html` lines 171–211 attach the idle reset to `mousemove`. Mobile users never trigger `mousemove`, so they see the idle text (`Wake / Up`) immediately on page load and it never resets.

**Recommended action:** Add `touchstart` and `scroll` listeners alongside `mousemove`:
```js
['mousemove', 'touchstart', 'scroll'].forEach(ev =>
  document.addEventListener(ev, resetTimer, { passive: true })
);
```

### `prefers-reduced-motion` for the terminal `delete` glitch effect

`scripts.js` `cmdDelete()` triggers a screen flash via `#glitch-overlay`. WCAG 2.3.1 caps flashes at 3 per second. The shake animation is fine. The flash overlay should be gated:

```js
if (window.matchMedia('(prefers-reduced-motion: reduce)').matches) return;
```

at the top of `cmdDelete()` (or skip just the overlay, keep the printed text).

### Cookie banner appears below all main content

`footer.html` lines 21–27 — banner is in the footer DOM. Visually positioned via CSS. If positioned at the bottom, ICO's "Accept and Reject equally easy" guidance is fine. If it covers content sticky-top, verify it can be dismissed. Worth a manual screenshot check on mobile.

### Footer GitHub link points to the personal username, not the project

Line 13: `<a href="https://github.com/JBowmanCODE">GitHub</a>`. Profile-level link is fine for personal branding, but a project link (`https://github.com/JBowmanCODE/johnb2026` or wherever this repo lives) would be a better trust signal for the "look, the source is available" angle.

If the repo is private, the profile link is fine — but then the YouTube key leak in §5.1 is less serious (still rotate it).

### News article count

5 articles now (added `ai-music-88-million-views.html` which I didn't fetch but exists in source). Pace is good. Confirm it's in `news-data.js`, `sitemap.xml`, and `llms.txt`.
