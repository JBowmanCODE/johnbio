## Executive Summary

**Overall Health Score: 7.2 / 10**

The site is genuinely impressive for a personal project — a comprehensive AI course, 20+ tools, structured news content, well-written legal pages, and a solid technical foundation in vanilla HTML/CSS/JS with Cloudflare Workers for API proxying. The fundamentals are strong. However, there are several issues that need addressing, two of which are significant enough to treat as urgent.

### Top 3 Urgent Issues

1. **🔴 GDPR Critical — Google Analytics fires without cookie consent.** The analytics tag loads immediately on every page with no cookie banner or consent mechanism. This is a breach of UK GDPR and PECR. It needs fixing before anything else.
2. **🟠 Security — HSTS is commented out and no CSP header.** HTTPS is in use but the Strict-Transport-Security header is disabled, and there is no Content-Security-Policy. These are standard hardening steps that are currently missing.
3. **🟠 Sitemap/robots.txt conflict.** Pages blocked in robots.txt (`/recipes`, `/wo`) also appear in sitemap.xml. This confuses crawlers and can harm indexing quality.

### Top 3 Quick Wins

1. Add a cookie consent banner (CMP) — fixes the most critical compliance issue in a day.
2. Enable the HSTS header in `.htaccess` (one line change, already written and commented out).
3. Remove `/recipes` and `/wo` from sitemap.xml since they're already disallowed in robots.txt (or flip the decision and allow them properly).

## 3. Accessibility (WCAG 2.1 / 2.2)

**Score: 6.5 / 10**

### What's Working Well
- `aria-label` attributes are used on interactive elements (hamburger button, back-to-top button, audio controls, search input).
- `aria-hidden="true"` is correctly applied to decorative icons.
- Images in article content have descriptive `alt` text.
- Mobile drawer uses `aria-hidden="true"` and has a proper close button with `aria-label`.
- Audio player controls have `aria-label` on all buttons.
- `role="slider"` and `aria-valuemin/max/now` on the audio progress bar.
- Language attribute `lang="en"` is present on the `<html>` element.
- `passive: true` on scroll event listeners (performance best practice).

### Issues Found

**Nav Dropdown Toggles Are Not Keyboard Accessible (High)**
The desktop navigation dropdowns use `<a class="nav-dropdown-toggle">` elements with no `href` attribute. These are not focusable by keyboard (Tab key will skip them). WCAG 2.1 Success Criterion 2.1.1 requires all functionality to be available via keyboard. Change these to `<button>` elements or add `href="#"` with `tabindex="0"` and `role="button"`.

**Inline onclick on About Button (Medium)**
`<button class="btn-primary" onclick="window.location.href='/about'">About Me</button>` — using `onclick` for navigation is bad practice. This should be an `<a>` element styled as a button, making it navigable via keyboard, middle-click, and screen readers.

**Focus Indicators Unknown (Medium)**
Cannot confirm without rendering the CSS, but many sites override the browser's default focus ring without providing a visible alternative. Check that all interactive elements have a clearly visible `:focus` and `:focus-visible` state that meets WCAG 2.1 AA contrast requirements.

**Audio Progress Slider Accessibility (Low-Medium)**
The audio progress bar uses a `<div>` with `role="slider"` and ARIA attributes — this is correct. However, `aria-valuenow` starts at 0 and should update dynamically as playback progresses. Verify the JavaScript updates this attribute as audio plays.

**No Skip Navigation Link (Medium)**
There is no "skip to main content" link at the top of pages. This is a WCAG best practice (SC 2.4.1) that lets keyboard and screen reader users skip past the navigation bar to the page content. Add a visually hidden but focusable `<a href="#main-content">Skip to content</a>` as the first element in the body.

**Course and Article Images: Lazy Loading Not Universal (Low)**
The article guide image uses `loading="lazy"` correctly, but not all images across the site can be confirmed to do so without reviewing all pages.

Move all Python scripts to a non-web-accessible directory, or add them to `.gitignore` and remove from the server.

```
# Header set Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
```
This is already written in `.htaccess` but commented out. Enable it. Once your HTTPS is confirmed stable, uncommenting this line takes 30 seconds and is a significant security and trust signal.

**Font Loading Strategy Could Be Improved (Low)**
Two separate Google Fonts `<link>` tags are used on most pages — one for the font families and one for Material Symbols. They could be combined, and adding `font-display: swap` via the URL parameter would prevent invisible text during font load (FOIT). Check if `&display=swap` is on both URLs.

Most CSS files use `?v=2` cache-busting (e.g., `styles.css?v=2`, `header.css?v=2`), but some page-specific CSS files like `index.css`, `about.css`, `legal.css` have no version parameter. If these files are updated, visitors may see cached old versions.

`.htaccess` sets `Header set X-XSS-Protection "1; mode=block"`. This header was deprecated and removed by Chrome in 2019. It is ignored by modern browsers. Remove it and replace with a proper `Content-Security-Policy` header instead.

## 5. Security Issues

**Score: 6.0 / 10**

### What's Working Well
- **Cloudflare Worker API proxy** — API keys for Anthropic, OpenAI, and Gemini are stored as Cloudflare Worker Secrets (environment variables), NOT in client-side code. This is correct and secure.
- **CORS origin allowlist** in the Worker — only `https://johnb.io` and `https://www.johnb.io` are whitelisted.
- **Rate limiting** — 20 requests per IP per day via Cloudflare KV. Good abuse protection.
- **Directory listing disabled** (`Options -Indexes` in .htaccess).
- **`.htaccess` self-protected** from direct access.
- **Firebase Security Rules** — privacy policy mentions rules restrict access to own data only. (Cannot verify without Firebase console access but the intent is correct.)
- **HTTPS enforced** — www → non-www redirect includes HTTPS.
- **`rel="noopener"` on all external links** — correctly prevents tab hijacking.

### Issues Found

**🔴 Worker Secret Hardcoded in Git Repository (High)**
In `workers/lm-chat-worker.js`:
```javascript
const WORKER_SECRET = 'cv-enhancer-internal-2026';
```
This shared secret (used for trusted worker-to-worker calls that bypass origin checking) is hardcoded in the file and committed to the Git repository. If the repository is public — or if it becomes public — this secret is exposed. Anyone who knows the secret can call your API worker without an origin check, bypassing the CORS protection and potentially exhausting your API credits.

**Recommended action:** Move `WORKER_SECRET` to a Cloudflare Worker Secret (environment variable), accessed via `env.WORKER_SECRET`, the same way the API keys are handled. Rotate the current value immediately if the repo is public or has ever been shared.

**🟠 HSTS Header Disabled (High)**
The Strict-Transport-Security header is commented out. This means browsers don't know to enforce HTTPS — they could be downgraded to HTTP on subsequent visits. Enable it:
```apache
Header set Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
```

**🟠 No Content-Security-Policy Header (Medium)**
No CSP header is set. A CSP is the primary defence against XSS attacks. Given the site runs user-supplied text through AI tools, and has a terminal-style UI component, a CSP is important. A starter CSP restricting `script-src` to `'self'` and your known CDNs would provide meaningful protection.

**🟠 error_log File in Webroot (Medium)**
The file `error_log` is in the webroot root directory and is publicly accessible. This file logs server-side PHP/application errors and may expose file paths, database connection strings, function names, or other internal details. This is an information disclosure risk.

**Recommended action:** Add to `.htaccess`:
```apache
<Files "error_log">
    Order allow,deny
    Deny from all
</Files>
```

**Firebase API Key in Client-Side JS (Low — Informational)**
`firebase-init.js` and `login.js` both contain the Firebase web API key (`AIzaSyB...`). This is expected behaviour — Firebase web API keys are intentionally public and security is enforced by Firebase Security Rules, not by key secrecy. However, confirm that your Firestore security rules properly restrict reads/writes to authenticated users only, and that there are no open rules in the Firebase console.

**No Permissions-Policy Header (Low)**
No `Permissions-Policy` header is set. This header controls what browser features (camera, microphone, geolocation) your site can use. While not critical, it's a standard hardening practice — particularly relevant for a site that uses browser APIs for text-to-speech and audio playback.

### Issues Found

**🔴 Google Analytics Fires Without Consent (Critical GDPR/PECR breach)**
On every page reviewed, Google Analytics fires immediately on page load:
```html
<script async src="https://www.googletagmanager.com/gtag/js?id=G-0J6PBWMRNY"></script>
```
There is no cookie consent banner, no CMP (Consent Management Platform), and no mechanism for users to opt in or out before analytics cookies are set. Under UK GDPR and PECR (Privacy and Electronic Communications Regulations), analytics cookies require **prior informed consent** from UK/EEA visitors. Loading GA unconditionally is a clear breach.

**Recommended fix options (choose one):**
- Use a free CMP like Cookiebot, CookieYes, or Osano. Add consent banner; fire GA only after consent.
- Switch to a privacy-first, consent-free analytics tool like Plausible or Fathom (no cookies, no GDPR consent needed — 100% compliant and much simpler).
- Use Google Analytics in `anonymize_ip` + consent mode, though this is more complex to implement correctly.

The simplest fix: add CookieYes (free tier) with GA4 consent mode integration. Takes about 2 hours.

**No Cookie Banner Anywhere on the Site (Critical — same issue)**
Despite the Privacy Policy correctly describing cookie use, there is no visible cookie consent UI anywhere in the codebase. The Privacy Policy section 9 says "You can disable cookies in your browser settings" — this is not a valid substitute for prior consent under PECR.

**Privacy Policy Not Linked from Footer (Medium)**
Could not verify footer content without rendering the page, but the Privacy Policy should be prominently linked from the site footer on every page. Check this is the case.

**Age Verification / COPPA/Under-18 Notice (Low)**
The Privacy Policy states the site is not directed at children under 13. However, the site hosts iGaming tools — bonus calculators, compliance maps, RG message generators. UK gambling regulations require clear age restrictions. Consider adding an explicit "18+" notice to iGaming tool pages and confirming the iGaming tools are for industry professionals only, not players.
