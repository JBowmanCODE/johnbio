## Executive Summary

**Overall Health Score: 7.5 / 10**

The site is well-built, modern, and clearly maintained by someone who understands web development. The vanilla HTML/CSS/JS approach keeps things fast and dependency-free, the legal pages are solid, security headers are comprehensive, and the structured data is ahead of most personal sites. The main gaps are in GEO/AEO optimisation, cookie consent implementation, accessibility refinements, and sitemap hygiene.

### Top 3 Urgent Issues

1. **Cookie consent does not gate Google Analytics** — GA fires before consent is granted (analytics_storage is set to `denied` by default, which is good, but the GA script itself still loads and the consent banner wording implies passive consent rather than active opt-in, which weakens GDPR compliance).

## 1. SEO (Search Engine Optimisation)

### 1.1 Title Tags and Meta Descriptions

**Issue:** Title tags are present and well-structured across all audited pages (Home, About, Contact, Privacy, Terms, News). Meta descriptions are present and within recommended character limits.

**Why it matters:** These are the primary elements shown in search results.

**Recommended action:** No immediate action needed. Consider A/B testing more compelling meta descriptions on key tool pages to improve CTR.

### 1.2 Heading Structure

**Issue:** The homepage uses a single `<h1>` ("AI Tools & Projects") which is correct. However, the hero H1 is split across two `<span>` elements and dynamically swapped via JS (idle state changes to "Wake Up"), which means search engines may see different text than users.

**Why it matters:** Google renders JS, but the dynamic swap could cause inconsistency in how the page title is indexed.

**Recommended action:** Ensure the default/initial H1 text is the canonical version you want indexed. Consider using a `<noscript>` fallback or keeping the H1 static.

### 1.3 Sitemap Issues

**Issue:** The sitemap.xml includes pages that should not be indexed:
- `header.html`, `footer.html`, `template.html` (partials/internal)
- `heart.html`, `wo.html`, `michael-scott.html` (disallowed in robots.txt)
- `cgi-bin/` (server directory)
- `verify.html`, `login.html` (utility pages)
- All URLs use `.html` extensions but .htaccess strips them — causing a redirect chain when crawlers follow sitemap URLs

**Why it matters:** Conflicting sitemap/robots.txt signals confuse crawlers. Redirect chains waste crawl budget and dilute link equity.

**Recommended action:**
- Remove all partial/internal pages from sitemap
- Remove pages disallowed in robots.txt
- Use clean URLs (without .html) in the sitemap to match the canonical URLs after rewrite
- Set differentiated `<priority>` values (homepage 1.0, tools 0.8, course lessons 0.6, legal pages 0.3)

### 1.4 Internal Linking

**Issue:** The homepage dynamically loads projects via JS (the `#projectsGrid` container is empty in the source HTML). This means the internal links to all 19+ tools are only available after JS execution.

**Why it matters:** While Google renders JS, links in the initial HTML are given more weight. Server-rendered links are more reliably crawled.

**Recommended action:** Consider adding a `<noscript>` fallback with static links to key pages, or rendering the project grid in the initial HTML.

### 1.5 Image Alt Text

**Issue:** The homepage and about page HTML contain no `<img>` tags in the source (images appear to be loaded dynamically or via CSS backgrounds). The footer banner image has good alt text. News card images use the post title as alt text, which is reasonable.

**Why it matters:** Alt text is critical for image SEO and accessibility.

**Recommended action:** Audit all dynamically loaded images to ensure alt text is programmatically applied. The about page profile image appears to be a CSS background-image — consider converting it to an `<img>` tag with descriptive alt text for SEO value.

### 1.6 URL Structure

**Issue:** Clean URLs are implemented via .htaccess rewrites (e.g., `/about` instead of `/about.html`). This is well done.

**Why it matters:** Clean URLs are better for SEO and user experience.

**Recommended action:** No action needed. This is correctly implemented.

### 1.7 Canonical Tags

**Issue:** Canonical tags are present on all audited pages and point to the correct clean URLs.

**Recommended action:** No action needed.

### 1.8 Blog/News Content Volume

**Issue:** The news section currently has only 3 articles. This is thin for SEO purposes.

**Why it matters:** Regular, high-quality content is one of the strongest SEO signals. Three articles is not enough to establish topical authority.

**Recommended action:** Aim for 2-4 new articles per month. See Section 9 for specific topic suggestions.

## 3. Accessibility (WCAG 2.1 / 2.2)

### 3.1 Skip Navigation

**Issue:** A skip-nav link (`<a href="#main-content" class="skip-nav">Skip to main content</a>`) is present in the header. The `main` element is dynamically assigned `id="main-content"` via JS.

**Why it matters:** Skip-nav is essential for keyboard users.

**Recommended action:** Add `id="main-content"` directly to the `<main>` element in HTML rather than relying on JS to set it. If JS fails, the skip link will be broken.

### 3.2 Focus Indicators

**Issue:** Good — the site has a global `:focus-visible` style (`outline: 2px solid #FF007F; outline-offset: 2px`). This is a solid implementation.

**Recommended action:** No action needed. This is well done.

### 3.3 ARIA Labels and Roles

**Issue:** Generally good. The hamburger menu button has an aria-label, modal dialogs have `role="dialog"` and `aria-label`, `aria-hidden` is toggled on open/close, and the `aria-expanded` attribute is used on dropdown toggles.

**Why it matters:** Proper ARIA usage is critical for screen reader users.

**Recommended action:** Minor — ensure `aria-expanded` is toggled programmatically when dropdowns open (verify this in scripts.js). The news filter group has `role="group"` with `aria-label`, which is correct.

### 3.4 Colour Contrast

**Issue:** The site uses a dark theme with the following notable colour combinations:
- Primary text: `#f8f5fd` on `#0e0e13` — contrast ratio ~18.5:1 (passes AAA)
- Variant text: `#acaab1` on `#0e0e13` — contrast ratio ~7.5:1 (passes AA)
- Some UI elements use `rgba(255,255,255,0.45)` which may fall below 4.5:1 on the dark background

**Why it matters:** Insufficient contrast is the most common WCAG violation.

**Recommended action:** Audit elements using low-opacity white text (e.g., the hero subtitle, footer copyright). Ensure all text meets at least 4.5:1 for normal text or 3:1 for large text.

### 3.5 Form Labels

**Issue:** The contact form has proper `<label>` elements linked to inputs via `for` attributes. Required fields are marked with `<span class="cf-required">*</span>`.

**Why it matters:** Form labels are essential for screen reader accessibility.

**Recommended action:** Consider adding `aria-required="true"` to required inputs for belt-and-braces screen reader support. The honeypot field correctly uses `aria-hidden="true"` and `tabindex="-1"`.

### 3.6 Missing `alt` Text on Dynamic Content

**Issue:** The project grid on the homepage is populated via JS. The news cards use image alt text derived from post titles (good). However, it is unclear if all dynamically generated images across tool pages have alt text.

**Recommended action:** Audit the `index.js` file to confirm alt text is set on all project card images. Audit individual tool pages for any images loaded dynamically without alt attributes.

## 4. Site Optimisation and Modern Web Practices

### 4.1 No jQuery or Legacy Frameworks

**Issue:** The site is built with vanilla HTML, CSS, and JavaScript. No jQuery, Bootstrap, or other legacy frameworks detected.

**Why it matters:** This is excellent for performance and maintainability.

**Recommended action:** No action needed. This is a strength.

### 4.2 Modern CSS

**Issue:** The site uses CSS custom properties (`:root` variables), modern selectors, flexbox, and grid. The code is clean and well-organised with separate CSS files per page.

**Recommended action:** No action needed.

### 4.3 Font Loading

**Issue:** Google Fonts are loaded with `<link rel="preconnect">` directives for both `fonts.googleapis.com` and `fonts.gstatic.com`, which is correct. However, the `styles.css` also imports fonts via `@import url(...)` at the top of the file — this creates a duplicate request and a render-blocking chain.

**Why it matters:** The `@import` in CSS is render-blocking and adds a sequential network request. The fonts are already loaded via `<link>` tags in the HTML.

**Recommended action:** Remove the `@import url(...)` line from `styles.css` since the fonts are already loaded via `<link>` tags in the HTML `<head>`. This will eliminate one render-blocking request.

### 4.4 Material Symbols Font

**Issue:** The Material Symbols Outlined icon font is loaded on every page. This is a large font file that includes thousands of icons when only a handful are used.

**Why it matters:** Loading the full icon font adds significant weight to every page load.

**Recommended action:** Consider switching to inline SVG icons for the ~15 icons actually used, or use the `text` parameter in the Google Fonts URL to subset only the icons you need (e.g., `&text=search,expand_more,arrow_upward,...`).

### 4.5 Header/Footer Loading via Fetch

**Issue:** The header and footer are loaded via `fetch()` calls in `scripts.js` and cached in `sessionStorage`. This is a clever approach for a static site but means the header/footer are not in the initial HTML.

**Why it matters:** This adds a FOUC (Flash of Unstyled Content) risk and means navigation links are not in the initial HTML for crawlers that don't execute JS.

**Recommended action:** This is a trade-off you have likely already considered. For a static site without a build tool, this is reasonable. If you ever adopt a build step (e.g., Eleventy, Astro), pre-rendering the header/footer would eliminate this issue.

### 4.6 Cache Busting

**Issue:** CSS files use query string versioning (e.g., `styles.css?v=2`, `header.css?v=4`). The .htaccess also supports filename-based versioning (`file.123.css`).

**Why it matters:** Query string cache busting works but is less reliable than filename-based versioning with some CDNs.

**Recommended action:** Consider migrating to filename-based versioning if you adopt a build tool. For now, the query string approach is functional.

### 4.7 Compression and Caching

**Issue:** The .htaccess enables gzip compression via `mod_deflate` for text-based resources and sets aggressive `Expires` headers (1 year for static assets, 1 month default). This is well configured.

**Recommended action:** No action needed. This is correctly implemented.

## 5. Security Issues

### 5.1 Firebase API Key in Client-Side JavaScript

**Issue:** The Firebase config (including `apiKey`) is exposed in `firebase-init.js` and duplicated in `login.js`.

**Why it matters:** While Firebase API keys are designed to be public (they identify the project, not authenticate it), an improperly secured Firebase project could be abused (e.g., excessive auth requests, Firestore reads/writes if rules are permissive).

**Recommended action:**
- Verify Firebase Security Rules are locked down (Firestore rules should restrict reads/writes to authenticated users' own data only)
- Enable Firebase App Check to prevent abuse from unofficial clients
- Set HTTP referrer restrictions on the API key in the Google Cloud Console
- Consider consolidating the Firebase config to a single file (firebase-init.js) and importing it in login.js to avoid duplication

### 5.2 Security Headers

**Recommended action:** No action needed. This is an excellent security configuration, significantly above average for a personal site.

### 5.3 Mixed Content

### 5.4 Error Log Exposure

**Why it matters:** Error logs can contain sensitive information.

### 5.5 Directory Listing

**Issue:** `Options -Indexes` is set in .htaccess, preventing directory listing. This is correct.

**Recommended action:** No action needed.

### 5.6 Content Security Policy

**Issue:** The CSP is comprehensive and includes `'unsafe-inline'` for both `script-src` and `style-src`. This is necessary for the current architecture (inline scripts and styles) but weakens the CSP.

**Why it matters:** `'unsafe-inline'` allows any inline script to execute, which is the main vector for XSS attacks.

**Recommended action:** Long-term, consider moving inline scripts to external files and using nonce-based CSP. For now, this is acceptable given the site's architecture.

**Issue:** A cookie banner exists in `footer.html` with Accept and Reject buttons. Google Analytics is loaded with `analytics_storage: 'denied'` by default, which is the correct consent-mode approach. However:
- The banner text says "by continuing to visit this site you agree to our use of cookies" — this is implied consent, which does not meet GDPR requirements for active opt-in
- The GA script tag loads on every page regardless of consent (even though storage is denied, the script still makes network requests to Google)
- It is unclear if the "Accept" button triggers `gtag('consent', 'update', { analytics_storage: 'granted' })` — this needs to be verified in the cookie handling code

- Replace with clear opt-in language: "We use cookies for analytics. You can accept or reject them."
- Ensure the GA script only loads after explicit consent (or continue using consent mode but remove the implied consent language)
- Verify that the Accept button correctly calls `gtag('consent', 'update', ...)` and that the Reject button prevents analytics cookies
