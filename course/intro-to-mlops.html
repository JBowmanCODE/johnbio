You build a model in a notebook. It works. You get 87% accuracy. You're happy. You deploy it to production. Three months later it's performing at 64%. You didn't touch it. The data changed. The users changed. The world changed. And now your model is broken in ways that are hard to debug because production is messier than your notebook.

The Gap Between Notebook and Production

A notebook is a controlled environment. You have a dataset. You split it into train and test. You train once, evaluate once, iterate. When you're done, you have a model file.

Production is chaos. Data arrives continuously. It's different than your training data. Your model makes predictions on it. Those predictions affect real decisions. Sometimes the predictions are wrong and you need to understand why.

Your notebook didn't have to handle new data every minute, different formats and distributions, multiple models running in parallel, rolling back to a previous model if the new one is bad, monitoring whether the model is still working, retraining when performance degrades, version control for data and models, compliance requirements, or budget constraints. A notebook solves one problem: build a model. Production needs to solve dozens more.

What MLOps Is and Why It Exists

MLOps is the infrastructure and practices that turn a model into a system that works reliably in production. It's borrowed from DevOps - the practices that make software deployable and maintainable. But ML adds complexity because your system has three moving parts: code, data, and models. In traditional software, the code is the source of truth. In ML, the code matters less than the data and the model weights.

MLOps exists because ML in production fails in different ways than regular software. A regular application can be fully tested before deployment. An ML system can never be fully tested - you'll always encounter data distributions you didn't see in training. A regular application is deterministic. An ML system is probabilistic - it makes mistakes, and those mistakes change over time as the data changes.

You need different tools and practices for this reality. See also the lessons on model deployment and monitoring and drift for where these practices get applied in practice.

The Key Stages: Data, Training, Deployment, Monitoring

Data management. You need to know what data your model was trained on. You need to track data quality. You need to catch when the data distribution shifts. You need to version datasets so that when a model breaks, you can recreate the exact conditions that created it. This is harder than it sounds because data is big and complex and it changes constantly.

Training. You need to be able to retrain models automatically. You need to track hyperparameters and results. You need to run experiments in parallel and compare them systematically. You need to version models and know which version is in production. This is infrastructure engineering - setting up systems that can reliably train, test, and version models at scale.

Deployment. You need to serve models with low latency and high availability. You need to be able to roll out new models gradually and roll back if something breaks. You need to handle multiple versions of a model running simultaneously. This is usually handled through containers and cloud infrastructure.

Monitoring. You need to track model performance on real data over time. You need to notice when performance degrades and alert humans. You need to log predictions so you can debug failures. You need to know when to retrain. This is ongoing - it's not a one-time step.

These stages feed back into each other. Monitoring tells you when to retrain. Training produces a new model. Deployment puts it in production. Monitoring checks if it works. The cycle continues.

Why ML in Production Fails Differently

A regular application usually fails because of bugs. The code has a flaw, you fix it, you deploy the fix. ML systems fail because the world changed.

Your model learned patterns from training data. In production, the data is different. Not in format - usually the same format. But the distribution is different. Users behave differently. Products change. Campaigns succeed or fail. The patterns the model learned aren't true anymore.

This is called drift and it's inevitable. You can't prevent it. You can only detect it and handle it. The next unit lesson on monitoring and drift goes into this in detail.

A regular application is also deterministic. You can test every branch. An ML model is probabilistic. You can't test every possible input. You don't know in advance what mistakes it will make. You can only observe mistakes in production and try to learn from them. This is why monitoring matters so much in ML - you can't catch all errors before deployment, you have to catch them after and respond.

Do Data Scientists Need to Care About MLOps?

Yes and no. At a small startup, probably not - you're moving fast and infrastructure is someone else's problem. At a large company with a mature ML practice, absolutely.

Data scientists should understand the basics. Not become MLOps engineers - that's a different skill. But you should understand how models go from notebook to production. You should understand what happens to your model when data distribution changes. You should know how to instrument your model so it's debuggable in production.

The companies that are good at ML are the ones where data scientists and MLOps engineers work together. The data scientist doesn't have to implement the monitoring system, but they should understand what needs to be monitored and why. They should think about edge cases that might not show up in training data. They should care that their model is reliable, not just accurate.

Do data scientists need to become infrastructure experts? No. Should they care that the model they build will actually work in the real world? Absolutely.

Frequently Asked Questions

What is MLOps?

MLOps is the infrastructure and practices that turn a trained machine learning model into a system that works reliably in production. It borrows from DevOps but adds complexity because ML systems have three moving parts - code, data, and model weights - rather than just code.

Why do ML models degrade in production?

ML models degrade because the world changes after training. User behaviour shifts, product ranges change, marketing campaigns alter patterns, and the data the model sees in production differs from the data it trained on. This is called drift and it is inevitable. The only question is how fast you notice and how you respond.

What are the main stages of MLOps?

The main stages are data management (tracking what data the model trained on, monitoring quality, versioning datasets), training (automated retraining, experiment tracking, model versioning), deployment (serving models with low latency, rolling out gradually, rollback capability), and monitoring (tracking performance on real data over time, alerting on degradation). These stages feed back into each other in a continuous cycle.

Do data scientists need to know MLOps?

Data scientists do not need to become MLOps engineers, but they should understand the basics. They should know how models go from notebook to production, what happens when data distribution changes, and how to instrument a model so it is debuggable in production. The companies that are good at ML are those where data scientists and MLOps engineers work together rather than in separate silos.

How It Works

An MLOps pipeline typically runs as a loop. Data arrives continuously from production systems. A data management layer validates quality and tracks distribution statistics. When enough new data has accumulated - or when monitoring detects degraded performance - a training job kicks off automatically.

The training job logs hyperparameters, results, and data versions so every trained model has full lineage. The new model is evaluated against the current production model. If it's better, it gets deployed - usually gradually, starting with a small percentage of traffic. Monitoring tracks its performance on real predictions.

Tools like MLflow handle experiment tracking. Docker containers standardise the serving environment. Cloud platforms like AWS SageMaker or GCP Vertex AI provide managed infrastructure. The monitoring layer closes the loop, triggering the next retraining when drift is detected.

Key Points
  • MLOps bridges the gap between a working notebook model and a reliable production system
  • ML systems have three moving parts - code, data, and model weights - making them harder to manage than regular software
  • ML models fail in production because the data distribution changes, not because the code breaks
  • The four main stages of MLOps are data management, training, deployment, and monitoring - in a continuous loop
  • Drift is inevitable - the goal is to detect it quickly and respond, not prevent it
  • Data scientists don't need to implement MLOps infrastructure, but they should understand what happens to their models after deployment
